## 一、总体设计思路

### 1.1 要解决什么问题

用户需要像 Google Docs 修订模式一样：**在保留 Markdown 结构的前提下**，对比两份文档、高亮变更，并能对**单个变更块**接受或拒绝，最终写回可编辑的 Markdown 字符串（`v-model` 双向绑定）。

纯文本行 diff 会把表格、列表、标题等结构打碎；纯 DOM diff 又无法无损回写 Markdown。因此采用 **「AST 定结构 + 文本 diff 定细节」** 的混合路线。

### 1.2 核心设计原则

| 原则 | 含义 |
|------|------|
| **语义优先** | 以 mdast 块（段落、标题、表格行、列表项等）为对齐单元，而不是按行或按字符切全文。 |
| **分层 diff** | 块级对齐 → 容器内递归（表格/列表/段落特例）→ 文本/行内词级或字符级高亮。 |
| **可交互契约** | 每个可操作的变更块对应一个 `DiffHunk`（双侧路径 + 旧/新节点快照），支持接受/拒绝。 |
| **纯函数合并 + 副作用在边界** | `buildMergedMdast` 无状态；写回 Markdown、重算 diff 由 Vue 层与 emit 完成。 |
| **安全默认可用** | 不可信 Markdown 经 sanitize 后再 `v-html`；diff 样式与工具栏在信任边界内注入。 |

### 1.3 端到端流水线（思路级）

```
oldMarkdown / newMarkdown
    → 解析为两份 mdast
    → 块级对齐，生成 merged mdast + hunks 索引
    → 递归合并「已对齐」节点（含表格列映射、inline 退化等特例）
    → 转为 HTML（高亮、工具栏、消毒）
    → 用户点击接受/拒绝 → 按策略改单侧或双侧 AST → 序列化回 Markdown → 触发重算
```

**心智模型**：diff 是**只读视图**；接受/拒绝是**对源 AST 的编辑**；UI 每次编辑后**整链重算**以保证一致（当前取舍，见下文演进）。

## 二、三大实现支柱（思路摘要）

### 2.1 块级对齐：加权匹配而非「能配就配」

**思路**：把旧、新文档的顶层（及递归子级）children 看成两条序列，找出一条「对齐路径」：哪些块视为同一块在修改（equal/modified），哪些整块新增或删除。

**与朴素 LCS 的差异（概念层）**：

- 不仅要求「类型/结构 key 能对上」，还要求**文本足够像**才允许配对（可配置阈值）。
- 配对时优先选**相似度更高**的对，避免「弱匹配抢占强匹配」（例如两个相似标题配错、真正该配对的段落被拆开）。
- 对标题、代码块、表格等**提高门槛**，减少短文本、重复结构上的误配。

**子级策略（不写公式，只写意图）**：

- 段落等：子节点结构剧变时（一边纯文本、一边带格式，或子节点数量极不对称）**退化为整段文本 diff**，避免 inline 子树 LCS 乱配。
- 代码块：用字符级 diff，并处理换行/缩进边界，避免高亮「绿块掉到下一行」。
- 唯一语言的代码块：用结构信号直接 1:1 配对，跳过相似度计算。

**详见**：`weighted-lcs-interview-answer.md`（面试展开用）。

---

### 2.2 表格：行对齐 + 列映射（两层不变量）

**思路**：表格变更分两类——**行的增删改** 与 **列的增删改/重命名**，用不同机制处理。

| 层级 | 做法 | 不变量 |
|------|------|--------|
| **整张 table** | 行序列做块级对齐；**仅用第一行表头** 计算「旧第几列 ↔ 新第几列」的全局映射，所有 equal 行共用。 | 列映射是**全表契约**。 |
| **单行 tableRow**（无 table 上下文时） | 按列下标 `i` 一对一合并单元格。 | 行内列下标是**局部契约**，禁止 LCS 乱配 cell。 |
| **单元格内** | 与普通段落相同，词级/结构级 diff。 | — |

**列映射意图**：

- 删除中间列时，右边列仍与旧表同名列对齐 diff，而不是与「错位的上一列」比。
- 表头 fuzzy 匹配（高阈值）支持**列重命名**；未匹配的新列/旧列分别标为新增/删除。
- 列顺序严重打乱时**降级为按位置硬配**，接受「像整列改写」的展示，避免算法死循环或错序。

**详见**：`table-column-mapping-interview-answer.md`。

---

### 2.3 接受 / 拒绝：双侧坐标 + 可配置写回策略

**思路**：每个 hunk 保存 **在新文档中的路径** 与 **在旧文档中的路径**（插入/删除时两侧下标推进方式不同），以及 **oldNode / newNode 快照**。点击接受或拒绝时，在对应侧的 AST 上执行「插入 / 删除 / 替换」，再按需序列化为 Markdown 写回。

**语义（产品层）**：

- **接受**：以新版本为准（默认改 old 侧，使旧稿追上 new）。
- **拒绝**：以旧版本为准（默认改 new 侧，使新稿回退到 old）。
- 可配置 **classic / syncBoth / mirror** 等，决定一次操作更新 old、new 或两侧。

**架构取舍**：

- 优先 **正确性与可推理性**：每次 resolve 后由父组件更新 Markdown，再 **全量 parse + merge + render**。
- hunk ID 在单次 build 内相对稳定（路径 + 结构签名），但 **resolve 后 path 会变**，外部若缓存「已审阅 id」需额外设计。

**详见**：`accept-reject-interview-answer.md`。

## 三、横切能力（与三大支柱并列）

### 3.1 渲染与安全

- **管线顺序**：mdast → HTML 树 → 解析内嵌原始 HTML → **消毒** → 代码高亮 → **注入 diff 包裹与工具栏** → 输出字符串。
- **原则**：用户内容必须先过白名单；工具栏等 UI 在消毒之后由可信代码生成。
- **展示**：块级变更用可包裹的容器 + 工具栏；表格行/单元格等不能外包 div 的节点用 class + 单元格内工具栏。

### 3.2 Vue 集成

- **Composable** 负责：解析、合并、防抖渲染 HTML；**组件** 负责事件委托（动态 HTML 上的接受/拒绝、快捷键）。
- **大对象**：最终 HTML 用浅响应式；AST 宜视为外部数据，避免深层 Proxy（文档与代码中仍有优化空间）。
- **可选共享**：`provide/inject` 让多面板共用一份 diff 状态，避免重复计算。

### 3.3 配置化边界

通过 `DiffConfig` 暴露业务可调参数，例如：块相似度阈值、表头 fuzzy 阈值、相似度计算最大文本长度、hunk 写回策略等——把「领域经验」从代码里抽到配置，而不是写死。

## 四、当前已落地的优化点

以下为**已在代码或配置中体现**的优化，按类别归纳。

### 4.1 对齐与展示质量

| 优化 | 作用 |
|------|------|
| 加权块级对齐 | 减少相邻相似块的错配，modified 更符合直觉。 |
| 分类型相似度阈值 | 标题/代码/表格等更严，段落等更松。 |
| 表格表头列映射 + 行 LCS | 删列、增列、列重命名场景可区分展示。 |
| 列映射非单调时位置回退 | 列对调时仍有确定性的降级结果。 |
| paragraph inline 结构退化 | 格式剧变时整段文本 diff，避免 inline LCS 噪声。 |
| 代码块字符 diff + 换行边界归一化 | 代码变更高亮不「挂」到下一行。 |
| 唯一 lang 代码块 1:1 匹配 | 多代码块文档中减少误配。 |
| 表格行/列表项级 hunk | 整行/整项可接受拒绝。 |
| 稳定 hunk ID（路径 + 结构签名） | 单次 session 内便于定位与测试。 |

### 4.2 性能与交互

| 优化 | 作用 |
|------|------|
| `maxSimilarityTextLength` 截断 | 限制相似度计算规模，避免超长段拖死主线程。 |
| HTML 渲染 debounce（约 120ms） | 连续输入时合并渲染，减轻 v-html 压力。 |
| 仅对需要更新的侧 `mdastToMarkdown` | resolve 时减少无谓序列化。 |
| 事件委托 | 动态 DOM 上不逐按钮绑监听，利于 v-html 整块替换。 |
| `hunkResolve` 预设 | 不同产品模式无需改核心逻辑。 |

### 4.3 安全与鲁棒性

| 优化 | 作用 |
|------|------|
| rehype-sanitize + 自定义 schema | 允许 diff 所需 class/属性，其余按 GitHub 风格收紧。 |
| 文本进 HTML 前转义 | 降低 diff 标记破坏 DOM 的风险。 |
| remark 容错解析 | 非法 Markdown 一般不抛错（但可能产生畸形 AST，见已知局限）。 |

## 五、已知局限与技术债（现状）

面试稿与代码审查中已识别的**非算法细节**问题，适合作为演进输入。

| 类别 | 现状 | 影响 |
|------|------|------|
| **全量重算** | 每次 accept/reject 后整链 parse → merge → render。 | 大文档卡顿；连续点击存在 props 未回流时的竞态风险。 |
| **AST 深拷贝** | 应用 hunk 时 JSON 克隆整树，且存在重复克隆。 | 大文档 apply 慢、内存峰值高。 |
| **Markdown 往返** | `mdastToMarkdown` 规范化（如列表符、格式）。 | 只改一处 hunk，emit 的全文可能与用户原文风格不一致。 |
| **hunk 粒度** | modified 多为整块容器；inline 细粒度不可单独接受。 | 无法「只接受段落里某一个词的修改」。 |
| **hunk ID** | 依赖 path，resolve 后漂移；无跨 rebuild 的语义 ID。 | 外部「已审阅集合」、协作同步难做。 |
| **表格** | 表头贪心、fuzzy 阈值固定经验值；列对调无 move 语义。 | 复杂表结构/列重排体验一般。 |
| **批量操作** | 无官方 batch API；多 hunk 需多次 emit + 重算。 | 批量接受性能差，path 需倒序等约定未封装。 |
| **防御分支** | 极少见的 text-text `mergeEqualNodes` 兜底拼接有误；`setTimeout` 未在卸载时清理。 | 边界场景潜在 bug / 轻微泄漏。 |
| **注释与实现** | 如 `shouldFlattenToTextDiff` 注释中的条件②未完全实现。 | 文档与行为略不一致。 |

## 六、后续演进路线

按 **「先止血 → 再体验 → 再规模 → 再协作」** 建议分期，可与 [`docs/TODO.md`](./TODO.md) 对齐排期。

### 阶段 A：正确性与稳定性（短期）

| 方向 | 思路 | 预期收益 |
|------|------|----------|
| resolve 内部状态 | old/new AST 在 composable 内维护「已应用」副本，不依赖父 props 同步回流再算下一 hunk。 | 消除连点竞态；批量操作可行。 |
| 批量 resolve API | 多 hunk 一次 apply，按 path **倒序**；一次 emit、一次 rebuild。 | 批量接受可用、性能线性化。 |
| hunk 失效检测 | apply 前校验定位节点与快照是否仍一致，失败则提示「变更已过期」。 | 多面板/异步改源时更安全。 |
| 生命周期 | debounce timer、`onScopeDispose` 清理。 | 无泄漏、路由切换更稳。 |
| 克隆优化 | `structuredClone` 或结构共享式 clone，去掉重复克隆。 | apply 延迟下降。 |

### 阶段 B：体验与粒度（中期）

| 方向 | 思路 | 预期收益 |
|------|------|----------|
| 增量 merge / 渲染 | resolve 后只重算受影响子树 + 局部更新 DOM，而非整页 v-html。 | 交互延迟体感明显改善。 |
| 双 ID | 语义 ID（内容 hash）+ 定位 ID（path）；rebuild 后映射已审阅状态。 | 变更列表、审计、协作基础。 |
| undo/redo | composable 内操作栈 + 快捷键。 | IDE 级修订体验。 |
| 局部接受 | inline / 子 hunk 树；段落级「全部接受 / 逐项接受」。 | 细粒度审阅。 |
| 表格增强 | 列内容指纹兜底；可选匈牙利匹配；列 move 语义。 | 复杂表、列重排可理解。 |
| 格式保真 | 可配置 `toMarkdown`、或基于 position 的局部 patch 写回。 | Git diff 噪声减少。 |

### 阶段 C：规模与性能（中长期）

| 方向 | 思路 | 预期收益 |
|------|------|----------|
| Worker diff | parse + `buildMergedMdast` 移入 Worker，主线程只渲染。 | 10 万字级文档可交互。 |
| 虚拟滚动 | 按顶层块切片渲染 HTML，视口外占位。 | 首屏与滚动流畅（需配合 Worker）。 |
| 分段 / 锚点 diff | 按 heading 切 section，先 section 对齐再块级 LCS。 | 降低单次 DP 规模。 |
| 对齐算法升级 | Patience / histogram 等降低误配与最坏复杂度。 | 大文档、高重复结构更稳。 |

### 阶段 D：产品与生态（长期）

| 方向 | 思路 | 预期收益 |
|------|------|----------|
| UX | 全部接受/拒绝、变更侧栏、上一条/下一条、筛选 insert/delete。 | 企业审阅流程。 |
| 三路合并 | base + old + new，冲突块类型。 | 分支合并场景。 |
| 语法扩展 | 脚注、数学、Mermaid 等结构化 diff 规则。 | 技术文档全覆盖。 |
| 与 OT/CRDT 边界 | 明确「离线批注 diff」vs「实时协同编辑」分工，必要时只输出 patch 不持编辑态。 | 架构边界清晰。 |

## 八、一句话总结

**MarkdownDiff** 用「**AST 管结构、加权对齐管块谁和谁一对、表格表头管列、文本 diff 管字、hunk 管可写回**」实现修订模式；当前在**对齐质量、表格、安全、防抖**上已有不少落地优化，但**全量重算、往返保真、hunk 粒度与 ID、大文档性能**仍是主要演进方向；后续宜先**稳住 resolve 状态与批量**，再**增量渲染与双 ID**，最后** Worker + 虚拟化与协作能力**。
